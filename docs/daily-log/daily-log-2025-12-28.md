## 2. LEARNING SUMMARY

### 2.1 Technical Learnings

#### Prisma ORM

**What Was Learned:**
- Prisma uses `schema.prisma` for declarative schema definition
- Models support native TypeScript types via `@prisma/client` generation
- `@@unique`, `@@index` for database-level constraints
- Enums defined at schema level become TypeScript enums
- `@relation` directive handles foreign keys automatically
- `Json` type for flexible metadata storage

**Why This Matters:**
- Type safety from database to application layer eliminates entire class of runtime errors
- Schema-as-source-of-truth enables easy migrations and rollback
- Generated Prisma Client includes autocomplete for all queries

#### bcrypt Password Hashing

**What Was Learned:**
- bcrypt with 10 salt rounds = industry standard for password storage
- `bcrypt.compare()` for verification (never decrypt, only compare)
- Hash length is always 60 characters regardless of input password length

**Why This Matters:**
- Never store plaintext passwords (security fundamental)
- bcrypt is slow-by-design to prevent brute force attacks
- 10 rounds balances security vs performance (~100ms per hash)

#### JWT Authentication

**What Was Learned:**
- Access tokens: short-lived (1h), carry user identity (sub, email, role)
- Refresh tokens: long-lived (7d), stored in database for revocation
- JWT structure: header, payload, signature (HMAC-SHA256)
- `fastify-jwt` plugin integrates with Fastify request/response lifecycle

**Why This Matters:**
- Stateless authentication scales horizontally (no session storage)
- Token revocation via database blacklist enables security without sacrificing performance
- Short access token lifespan limits damage from token leakage

#### Fastify Plugin System

**What Was Learned:**
- Fastify uses `register()` for plugin encapsulation
- Plugins can be scoped (prefix support)
- `fastify.decorate()` extends instance with custom properties
- `onRequest` hooks run before route handlers

**Why This Matters:**
- Encapsulation prevents plugin conflicts
- Decorators enable clean authentication (`fastify.authenticate`)
- Hooks enable cross-cutting concerns (logging, auth) without route duplication

### 2.2 Architectural Learnings

#### Layered Architecture Reinforcement

**Observation:**
```
Routes (HTTP layer) ‚Üí Services (business logic) ‚Üí Database Helpers ‚Üí Prisma Client
```

**Why This Matters:**
- Services contain pure business logic (testable without HTTP)
- Routes handle only HTTP concerns (parsing, response formatting)
- Database helpers abstract Prisma queries (swap ORM without touching services)

- Complex queries documented in one place

**Why This Matters:**

### 2.4 Engineering Best Practices Reinforced

#### Dependency Management

**Practice:**
- Add dependency to `package.json` before using it
- Include both runtime and `@types/*` dependencies
- Use lockfile (`package-lock.json`) for reproducible builds

**Why This Matters:**
- Ensures code runs in production same as development
- Prevents "works on my machine" issues
- Security updates tracked via `npm audit`

#### File Naming Conventions

**Practice:**
- TypeScript: `camelCase.ts` (flight.service.ts)
- Types: `PascalCase` (Flight, UserFlight)
- Routes: `*.routes.ts` suffix
- Utils: `*.ts` with `index.ts` barrel export

**Why This Matters:**
- File purpose immediately clear from name
- Easy to locate files via IDE fuzzy search
- Consistency reduces cognitive load

#### Error Handling Patterns

**Practice:**
```typescript
if (error instanceof ZodError) {
  // Validation error - return 400 with details
}
if (error instanceof AuthError) {
  // Auth-specific error - use status code from error
}
// Fallback - 500 with generic message
```

**Why This Matters:**
- Users receive actionable error messages
- Sensitive details not leaked in production
- Debugging information preserved in development

## 7. DECISION LOG

### 7.1 Technical Decisions Made Today

| Decision | Rationale | Alternatives Considered |
|----------|-----------|------------------------|
| Prisma over TypeORM | TypeScript-first, better DX | TypeORM, Drizzle, raw SQL |
| bcrypt over argon2 | Battle-tested, simple API | argon2 (newer, more complex) |
| JWT over sessions | Stateless, scales horizontally | Sessions, cookies |
| Access token 1h expiry | Balance security vs UX | 15min (too annoying), 24h (too risky) |
| Refresh token 7d expiry | Reduce re-login frequency | 30d (too risky), 1d (too annoying) |
| Pino over Winston | Faster, structured JSON | Winston (slower) |
| Zod over Joi | TypeScript-first | Joi, Yup |

### 7.2 Architectural Decisions Made Today

| Decision | Rationale |
|----------|-----------|
| Store refresh tokens in DB | Enable revocation (logout, password change) |
| Per-flight notification prefs | Granular control = better UX |
| Audit trail from day 1 | Debugging production issues |
| Connection risk enum in DB | Queryable, no magic strings |
| UserRole enum for RBAC | Support PREMIUM tier from start |
